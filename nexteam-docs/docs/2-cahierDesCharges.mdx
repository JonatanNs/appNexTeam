# Cahier des charges

## 1 -Présentation du projet

Plateforme intranet permettant la communication et la gestion interne des employés. NexTeam répond à un besoin majeur rencontré dans de nombreuses entreprises : le manque de communication et de centralisation des échanges entre les employés, les ressources humaines, les responsables et les managers.

L’application NexTeam permet de pallier cette problématique grâce à un système de messagerie intégré favorisant les échanges rapides et directs entre les différents acteurs de l’entreprise. La plateforme permet également de centraliser plusieurs outils internes au sein d’une seule application afin de simplifier l’organisation et améliorer la productivité des équipes.

NexTeam propose notamment :

- une messagerie interne en temps réel.
- un système de gestion des tâches quotidiennes.
- un espace d’actualités et de communication interne.
- une gestion des rôles et permissions utilisateurs.
- une carte employé numérique accessible depuis l’application.
- Recevoir ses fiches de paye et ses avantages.

L’objectif du projet est de créer une solution moderne, intuitive et sécurisée permettant d’améliorer la collaboration, la circulation des informations et le suivi des activités au sein d’une entreprise.

---

### Problématique

De nombreuses entreprises utilisent plusieurs outils distincts pour gérer la communication interne, les tâches des employés et les échanges avec les ressources humaines. Cette dispersion des outils peut compliquer l’organisation, ralentir les échanges et réduire la productivité des équipes.

NexTeam a pour objectif de centraliser ces fonctionnalités au sein d’une plateforme unique, moderne et accessible.

## 4 - Maquettes

Avant de commencer le développement d'une application, il est recommandé de passer par plusieurs étapes de conception visuelle. Ces étapes permettent de valider les besoins, améliorer l'expérience utilisateur et réduire les erreurs de développement.

**Pourquoi réaliser des maquettes ?**

La création de maquettes présente plusieurs avantages :

- **Visualiser le produit avant le développement**.
- **Valider les besoins fonctionnels** avec les parties prenantes.
- **Identifier les problèmes d'ergonomie** dès les premières phases du projet.
- **Faciliter les échanges** entre les équipes métier, UX/UI et développement.
- **Réduire les coûts de modification** en détectant les erreurs avant l'implémentation.
- **Servir de référence** tout au long du développement.

---

### Wireframe

Le wireframe représente la structure générale d'une interface sans se concentrer sur l'aspect graphique.

**Objectifs** :

- Définir l'organisation des éléments.
- Structurer les parcours utilisateurs.
- Valider l'emplacement des fonctionnalités.
- Se concentrer sur l'expérience utilisateur avant le design.

## 9 - Plan de tests

**Objectif** :

L’objectif des tests est de valider :

- le bon fonctionnement des règles métier
- la robustesse du système face aux cas nominaux et cas d’erreurs
- l’intégration correcte entre les couches (API → Service → Repository → DB)

Les tests sont divisés en deux niveaux :

- **Tests unitaires** : validation isolée de la logique métier (Mockito, sans Spring context complet)
- **Tests d’intégration** : validation du comportement global (API REST + Spring Boot + DB)

---

### 3. ⚠️ Cas d’erreurs globaux

Ces cas doivent être testés sur l’ensemble des API.

| Code HTTP | Signification | Cas typique | Résultat attendu |
|----------|--------------|-------------|-----------------|
| 200 | OK | Requête réussie (GET, PUT, DELETE logique) | Réponse valide retournée |
| 201 | Created | Création d’une ressource (POST) | Ressource créée + payload retourné |
| 204 | No Content | Suppression réussie | Aucune réponse retournée |
| 400 | Bad Request | Données invalides / validation échouée | Erreur de validation retournée |
| 401 | Unauthorized | Utilisateur non authentifié | Accès refusé |
| 403 | Forbidden | Utilisateur sans permission | Accès interdit |
| 404 | Not Found | Ressource inexistante | Ressource introuvable |
| 409 | Conflict | Conflit métier (ex: email déjà utilisé) | Erreur de conflit |
| 500 | Internal Server Error | Erreur serveur inattendue | Erreur technique loggée |

## 11 - Conclusion

Ce projet a pour objectif de concevoir une plateforme collaborative interne permettant d’améliorer la communication, l’organisation et la gestion des tâches au sein de l’entreprise.

Grâce aux différentes fonctionnalités mises en place selon les rôles des utilisateurs (Employé, Manager, RH et Administrateur), la plateforme permet de centraliser les échanges, le suivi des tâches ainsi que la diffusion des informations importantes de l’entreprise.

L’analyse des besoins fonctionnels et la modélisation UML réalisées dans ce cahier des charges constituent une base solide pour le développement de l’application, aussi bien sur la partie frontend que backend.

L’objectif final est de proposer une solution moderne, sécurisée et intuitive, capable de faciliter la collaboration entre les différents acteurs de l’entreprise tout en assurant une gestion efficace des données et des accès.
